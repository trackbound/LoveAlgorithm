# 아트 리소스 네이밍 컨벤션 (BG / Character / SD / CG)

> **Single Source of Truth.** `Assets/Resources/` 하위의 모든 아트 리소스 PNG 파일명은 이 문서를 따른다.
> 코드/CSV에서 리소스를 참조하는 모든 경로는 이 컨벤션에 의해 결정된다.
>
> 변경 이력: 기존 PascalCase + 의미 단어 조합 → **소문자 + 숫자 ID 기반** 으로 일괄 전환.

---

## 0. 설계 원칙

1. **AI/스크립트 파싱 친화** — 모든 파일명은 단일 정규식으로 카테고리/소속/인덱스를 추출 가능해야 한다.
2. **확장 시 코드 무수정** — 새 캐릭터/장소/감정 추가 시 파일만 추가하면 동작. enum/하드코딩 금지.
3. **사람 가독성은 별도 메타로** — 의미 단어(SeoDaEun, BrightSmile 등)는 파일명에서 제거하고 본 문서의 매핑 테이블과 Excel 시트로만 관리.
4. **소문자 + 언더스코어** — 모든 파일명은 `^[a-z0-9_]+\.png$` 패턴만 허용.
5. **고정 자리수** — ID는 항상 2자리 (`01`, `_10`). 한 자리 사용 금지.

---

## 1. 공통 형식

```
{prefix}_{...fields}_{index}.png
```

| Prefix | 카테고리 | 폴더 |
|--------|---------|------|
| `bg_` | 배경 (Background) | `Assets/Resources/Backgrounds/` |
| `c` (prefix 없음, 바로 캐릭터 ID) | 캐릭터 표정 (Character Emotion) | `Assets/Resources/Characters/` |
| `sd_` | SD 일러스트 | `Assets/Resources/SD/` |
| `cg_` | CG 일러스트 | `Assets/Resources/CG/` |

> **주의**: Character 카테고리만 `bg_/sd_/cg_` 같은 카테고리 prefix 없이 캐릭터 ID(`c01`~)로 시작한다. 짧고 빈번히 호출되는 리소스라 prefix를 생략한다.

## 2. BG (배경)

### 형식
```
bg_{장소ID}_{장소키}_{인덱스}.png
```
- `장소ID`: 2자리 숫자 (장소 매핑 테이블 참조)
- `장소키`: 영소문자 (장소 식별용 의미 단어, 작가/AI 가독성용)
- `인덱스`: 2자리 (해당 장소 내 일련번호, `01`부터 시작)

> SD/CG와 동일하게 **prefix → 숫자 ID → 영문 키 → 인덱스** 4필드 구조를 따른다.
> `장소ID`와 `장소키`는 1:1 고정 매핑이며, 둘 다 파일명에 표시해 정렬과 가독성을 동시에 확보한다.

### 정규식
```
^bg_(?P<place_id>\d{2})_(?P<place_key>[a-z]+)_(?P<idx>\d{2})\.png$
```

### 장소 ID / 키 매핑

| ID | 장소키 | 장소 | 비고 |
|----|--------|------|------|
| `00` | `common` | 특수/공통 | 검은화면 등 장소 무관 컷 |
| `10` | `room` | 자취방 | MyRoom |
| `20` | `eng` | 공대 | Engineering |
| `30` | `campus` | 캠퍼스 | Campus street |
| `40` | `sc` | 학생회관 | Student Center |
| `50` | `lec` | 교양관 | Lecture hall |
| `60` | `conv` | 편의점 | Convenience store |
| `70`~ | (예약) | | 10 단위로 증가 |

### 예시
- `bg_00_common_00.png` — 검은 화면
- `bg_10_room_01.png` — 자취방 전경 낮
- `bg_10_room_06.png` — 자취방 책상
- `bg_20_eng_05.png` — 공대 강의실 낮
- `bg_30_campus_01.png` — 캠퍼스거리1 맑음
- `bg_40_sc_06.png` — 학생회관 동방 낮 나무
- `bg_50_lec_01.png` — 교양관 복도 낮
- `bg_60_conv_01.png` — 편의점 앞 밤
- `bg_60_conv_02.png` — 편의점 안

### 확장 절차
1. 새 장소는 `60`, `70`... 으로 부여하고 **장소키**도 함께 결정 (3~6자 영소문자).
2. 본 문서 장소 ID/키 매핑 테이블에 추가.
3. `BG_List.xlsx`에 새 시트 추가.
4. 한 ID에 키 하나만 허용 — 변경 시 일괄 rename 필요.

## 3. Character (캐릭터 표정)

### 형식
```
c{캐릭터ID}_{감정ID}.png
```
- `캐릭터ID`: 2자리 (캐릭터 매핑 테이블 참조)
- `감정ID`: 2자리 — `{유형}{변형}` (유형 1자리 + 변형 1자리)

### 정규식
```
^c(\d{2})_(\d{2})\.png$
```

### 캐릭터 ID 매핑

| ID | 이름 | 영문 (참고) |
|----|------|-------------|
| `c01` | 로아 | Roa |
| `c02` | 서다은 | SeoDaEun |
| `c03` | 하예은 | HaYeEun |
| `c04` | 도희원 | DoHeewon |
| `c05` | 이봄 | LeeBom |
| `c06`~ | (예약) | |

### 감정 ID 매핑

| 유형 (1자리) | 의미 | 변형 1 | 변형 2 | 변형 3 | 변형 4 | 변형 5 |
|--------------|------|--------|--------|--------|--------|--------|
| `0` | 기본 | `_00` 기본 | | | | |
| `1` | 희 (기쁨) | `_11` 눈웃음 | `_12` 밝게웃음 | `_13` 활짝 | `_14` 행복 | `_15` (예비) |
| `2` | 로 (분노) | `_21` 찌릿 | `_22` 삐짐 | `_23` 머쓱 | `_24` 어질어질 | `_25` (예비) |
| `3` | 애 (슬픔) | `_31` 울먹 | `_32` 눈물 주르륵 | `_33` 와아앙 울기 | `_34` 부끄러워 | `_35` 피곤/졸려 |
| `4` | 락 (놀람/즐거움) | `_41` 깜짝 | `_42` 반짝빤짝 | `_43` 궁금 | `_44` 윙크 | `_45` 자신만만 |
| `5`~ | (예약) | | | | | |

### 예시
- `c01_00.png` — 로아 기본
- `c01_12.png` — 로아 밝게웃음
- `c03_34.png` — 하예은 부끄러워

### 확장 절차
- **새 캐릭터**: 다음 번호(`c06`...) 부여 후 `Character_Emotion_List.xlsx`에 새 시트 추가.
- **새 감정 유형**: `5`~ 부여 후 본 문서의 감정 ID 매핑 테이블 갱신.
- **기존 유형 내 변형 추가**: `_15`, `_25` 등 예비 슬롯 사용.

## 4. SD

### 형식
```
sd_{넘버링}_{대분류}_{소분류}_{인덱스}.png
```
- `넘버링`: 2자리, SD 전체 통합 일련번호 (스프레드시트 No. 열과 동일)
- `대분류`: 분류 키 (영소문자, 매핑 테이블 참조)
- `소분류`: 대분류 내 키 (영소문자, 매핑 테이블 참조)
- `인덱스`: 2자리, 같은 (대분류, 소분류) 내 변형 일련번호

### 정규식
```
^sd_(\d{2})_([a-z]+)_([a-z]+)_(\d{2})\.png$
```

### 대분류 / 소분류 매핑

| 대분류 | 의미 | 소분류 예시 |
|--------|------|-------------|
| `char` | 캐릭터 SD | `roa`, `seodaeun`, `hayeeun`, `doheewon`, `leebom` |
| `event` | 이벤트 SD | `festival`, `mt`, `confess` 등 |
| (확장) | | 추가 시 본 문서 갱신 |

**소분류(char) = 캐릭터 영문 키** — 캐릭터 ID 매핑 테이블의 영문(Roa, SeoDaEun…) 을 소문자로 사용.

### 예시
- `sd_01_char_seodaeun_01.png` — 다은 첫만남
- `sd_02_char_doheewon_01.png` — 희원 첫만남
- `sd_03_char_leebom_01.png` — 봄 첫만남
- `sd_10_event_festival_01.png` — 축제 이벤트 SD 1번 (예시)

### 확장 절차
1. `SD_List.xlsx`에 새 No.로 행 추가, 리소스명/설명/작업 채움.
2. 새 대분류·소분류 도입 시 본 문서 매핑 테이블 갱신.

---

## 5. CG

### 형식
```
cg_{넘버링}_{대분류}_{소분류}_{인덱스}.png
```
- 규칙은 SD와 동일. prefix만 `cg_`.

### 정규식
```
^cg_(\d{2})_([a-z]+)_([a-z]+)_(\d{2})\.png$
```

### 대분류 / 소분류 매핑

SD와 동일한 매핑을 공유한다 (`char`, `event`...).

### 예시
- `cg_01_char_roa_01.png` — 로아 첫만남
- `cg_02_char_hayeeun_01.png` — 예은 입부신청서 작성

### 확장 절차
1. `CG_List.xlsx`에 새 No.로 행 추가.
2. 새 대분류·소분류 도입 시 본 문서 매핑 테이블 갱신.

## 6. 폴더 구조

```
Assets/Resources/
├── Backgrounds/        # bg_*.png (현재 평면, 필요 시 장소별 하위 폴더화 검토)
├── Characters/         # c01_*.png ~ c05_*.png (현재 평면)
├── SD/                 # sd_*.png
└── CG/                 # cg_*.png
```

> 현재는 모두 평면(flat) 구조. 카테고리 prefix가 파일명에 포함되어 충돌이 없으므로 폴더 분리만 유지하면 충분하다.
> 리소스가 100개 이상으로 늘어나면 `Backgrounds/_10_room/` 같이 ID prefix 기반 서브폴더 도입 검토.

## 7. AI / 스크립트용 파싱 요약

다음 단일 정규식 세트로 모든 리소스를 카테고리·메타 분해 가능:

```python
PATTERNS = {
    "bg":   r"^bg_(?P<place_id>\d{2})_(?P<place_key>[a-z]+)_(?P<idx>\d{2})\.png$",
    "char": r"^c(?P<char>\d{2})_(?P<emotion>\d{2})\.png$",
    "sd":   r"^sd_(?P<no>\d{2})_(?P<major>[a-z]+)_(?P<minor>[a-z]+)_(?P<idx>\d{2})\.png$",
    "cg":   r"^cg_(?P<no>\d{2})_(?P<major>[a-z]+)_(?P<minor>[a-z]+)_(?P<idx>\d{2})\.png$",
}
```

ID → 사람 가독 이름 변환은 본 문서의 매핑 테이블 또는 Excel 시트에서 조회.

## 8. 변경/삭제 시 체크리스트

리소스 추가·이름 변경·삭제 시:

- [ ] PNG 파일 + `.meta` 동시 처리 (Unity GUID 보존)
- [ ] 해당 카테고리 Excel(`BG_List.xlsx` / `Character_Emotion_List.xlsx` / `SD_List.xlsx` / `CG_List.xlsx`) 갱신
- [ ] 정규식 매칭 확인 (위 PATTERNS)
- [ ] 참조하는 CSV 스크립트 (`Assets/Resources/Story/*.csv`)의 캐릭터/배경/CG 컬럼 갱신
- [ ] 코드 내 하드코딩된 리소스 경로 검색 (`Resources.Load` 호출부) — 가능하면 SO/CSV로 외부화
- [ ] `docs/ASSET_REQUESTS.md` 의 해당 항목 상태 갱신

## 9. 위반 처리

- 신규 파일이 정규식에 매칭되지 않으면 임포트 단계에서 거부 (에디터 스크립트 — 추후 구현).
- 매핑 테이블에 없는 ID(`c99_*`, `bg_99_*`)는 임시 placeholder 로만 사용. PR/커밋 전 정규 번호로 교체.
